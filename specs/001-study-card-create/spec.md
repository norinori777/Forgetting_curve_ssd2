# Feature Specification: 学習カード登録画面

**Feature Branch**: `001-study-card-create`  
**Created**: 2026-03-21  
**Status**: Draft  
**Input**: User description: "学習カード登録画面を作成する。ヘッダーメニューに学習カード登録のメニューを追加する。学習カードが登録できること。学習カードには、タイトルが登録できること。学習カードには、学習内容が登録できること。学習カードには、タグが登録できること。学習カードには、コレクションが登録できること。学習一覧Pageの右上に学習カード登録ボタンを設けて、学習カード登録画面に遷移する。"

## Clarifications

### Session 2026-03-21

- Q: タグ入力で未登録のタグ名が入力された場合はどう扱う？ → A: 入力したタグ名をそのまま受け付け、未登録タグは新規作成してカードに紐づける
- Q: 登録成功後、ユーザはどこへ遷移するべきか？ → A: 登録成功後はカード一覧ページへ遷移し、成功メッセージを表示する
- Q: コレクションは登録画面でどう扱うべきか？ → A: コレクションは既存候補から選択のみとし、新規作成はこの画面では行わない
- Q: 具体的なメッセージ定義はどこで管理するべきか？ → A: 共通仕様として docs/messages.md を作成し、各画面仕様から参照する
- Q: docs/messages.md ではどの種類の文言を管理するべきか？ → A: 成功・失敗・バリデーション・補助文言などのメッセージだけを docs/messages.md で管理する

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 学習カードを新規登録する (Priority: P1)

ユーザは、専用の登録画面でタイトル、学習内容、タグ、コレクションを入力し、新しい学習カードを登録したい。

**Why this priority**: 学習カードを追加できないと、その後の一覧表示、復習、統計のすべてが成立しないため、最優先の基本機能である。

**Independent Test**: 登録画面を開き、必須項目を入力して登録し、新しいカードが保存済みデータとして扱われることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが学習カード登録画面を開いている, **When** タイトルと学習内容を入力して登録する, **Then** 新しい学習カードが登録される
2. **Given** ユーザがタグを入力して登録する, **When** 登録が完了する, **Then** 入力したタグがそのカードに紐づく
3. **Given** ユーザがコレクションを選択して登録する, **When** 登録が完了する, **Then** 選択したコレクションがそのカードに紐づく
4. **Given** ユーザがタイトルまたは学習内容を未入力のまま登録しようとする, **When** 登録操作を実行する, **Then** 必須項目が不足していることが分かり、登録は完了しない
5. **Given** ユーザが学習カードの登録を完了した, **When** 登録成功処理が終わる, **Then** カード一覧ページへ遷移し、成功メッセージが表示される

---

### User Story 2 - 一覧やヘッダーから登録画面へ移動する (Priority: P2)

ユーザは、カード一覧ページやヘッダーメニューから迷わず登録画面へ移動したい。

**Why this priority**: 新規カード作成の導線が分かりづらいと、登録機能があっても利用されにくいためである。

**Independent Test**: ヘッダーメニューとカード一覧ページ右上の両方から登録画面へ移動でき、戻る導線も確認できる。

**Acceptance Scenarios**:

1. **Given** ユーザがどのトップレベルページにいても, **When** ヘッダーメニューの「学習カード登録」を選ぶ, **Then** 学習カード登録画面へ移動できる
2. **Given** ユーザがカード一覧ページを見ている, **When** 右上の「新規カード」を選ぶ, **Then** 学習カード登録画面へ移動できる
3. **Given** ユーザが学習カード登録画面を見ている, **When** 「一覧へ戻る」を選ぶ, **Then** カード一覧ページへ戻れる
4. **Given** ユーザが登録を成功させてカード一覧ページへ戻った, **When** 一覧画面が表示される, **Then** 登録成功メッセージによって完了を認識できる

---

### User Story 3 - 入力内容を確認して調整する (Priority: P3)

ユーザは、登録前に入力内容を見直し、必要に応じて修正やリセットを行いたい。

**Why this priority**: 誤登録を減らし、入力途中でも内容を整理しやすくすることで、登録体験を安定させるためである。

**Independent Test**: タイトル、学習内容、タグ、コレクションを入力し、プレビュー確認と入力リセットが行えることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが登録内容を入力している, **When** 画面内の確認領域を見る, **Then** 現在のタイトル、タグ、コレクション、学習内容の要点を確認できる
2. **Given** ユーザが入力途中である, **When** 「入力をリセット」を選ぶ, **Then** 入力済み内容が初期状態に戻る
3. **Given** ユーザが入力途中である, **When** 画面を離れようとする, **Then** 未保存の内容が失われる可能性を認識できる

### Edge Cases

- タイトルだけ、または学習内容だけを入力した状態では登録できない
- タグを空欄のまま登録してもカード登録自体はできる
- 未登録のタグ名を入力した場合でも、そのタグ名で新しいタグを作成して登録できる
- コレクション未選択でもカード登録自体はできる
- ネットワーク失敗や一時的な API エラー時でも、入力済み内容を保持したまま再試行できる
- 利用可能なコレクションが存在しない場合でも、コレクション未設定で登録できる
- 登録画面ではコレクションの新規作成はできず、既存候補がない場合は未選択のまま登録する
- 非常に長い学習内容でも、内容全体を入力して確認できる
- 登録に失敗した場合、入力済み内容を失わずに再試行できる
- 登録成功後にカード一覧へ戻った際、成功したことが明示される
- 具体的な表示メッセージは画面仕様に直書きせず、共通のメッセージ一覧を参照して定義する
- 見出し、ボタン名、ラベル、プレースホルダーは画面仕様側に残し、共通メッセージ一覧には含めない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 学習カードを新規登録するための専用ページを提供する
- **FR-002**: System MUST ヘッダーメニューに「学習カード登録」への導線を表示する
- **FR-003**: System MUST カード一覧ページの右上に「新規カード」導線を表示する
- **FR-004**: System MUST 学習カード登録ページでタイトルを入力できるようにする
- **FR-005**: System MUST 学習カード登録ページで学習内容を入力できるようにする
- **FR-006**: System MUST 学習カード登録ページでタグを入力できるようにする
- **FR-007**: System MUST 学習カード登録ページで既存候補からコレクションを選択できるようにする
- **FR-007a**: System MUST 学習カード登録ページではコレクションの新規作成を行わない
- **FR-008**: System MUST タイトルと学習内容を必須項目として扱う
- **FR-009**: System MUST 登録完了時に、入力したタイトル、学習内容、タグ、コレクションを持つ学習カードを保存対象として扱う
- **FR-010**: System MUST タグを複数入力できるようにし、ユーザが複数タグであることを理解できる入力補助を表示する
- **FR-010a**: System MUST 未登録のタグ名が入力された場合、そのタグを新規作成してカードに紐づける
- **FR-011**: System MUST コレクションを未選択のままでもカード登録を完了できる
- **FR-012**: System MUST 登録画面内に、現在入力している主要項目を確認できるプレビュー領域を表示する
- **FR-013**: System MUST 登録画面に「登録する」「入力をリセット」「一覧へ戻る」の操作を表示する
- **FR-014**: System MUST 必須項目が不足している場合、ユーザに不足内容が分かる状態で登録を止める
- **FR-015**: System MUST 登録に失敗した場合、ユーザが原因を認識し、入力済み内容を失わずに再試行できるようにする
- **FR-015a**: System MUST 登録成功後はカード一覧ページへ遷移し、登録成功メッセージを表示する
- **FR-016**: System MUST 学習カード登録ページの情報配置と主要UI構成を、以下の画面イメージに整合させる。ただし、画面イメージに含まれる下書き保存は将来拡張の参考要素であり、本機能の必須実装対象には含めない
- **FR-017**: System MUST 学習カード登録ページで、PC表示とモバイル表示の双方において、タイトル、学習内容、タグ、コレクション、主要操作の順で認識できるように配置する。モバイル表示では主要操作が縦方向に連続して到達可能であること
- **FR-018**: System MUST 画面で使用する具体的なユーザー向けメッセージを共通仕様として docs/messages.md で管理し、画面仕様から参照できるようにする
- **FR-018a**: System MUST docs/messages.md では成功、失敗、バリデーション、補助文言を管理し、見出し、ボタン名、ラベル、プレースホルダーは画面仕様側で管理する

### Key Entities *(include if feature involves data)*

- **Learning Card**: タイトル、学習内容、タグ、コレクションを持つ学習対象
- **Tag**: 学習カードを分類するための任意のラベル。1つのカードに複数付与できる
- **Collection**: 学習カードをまとめる単位。カードは未所属でもよい
- **Learning Card Draft**: 登録画面でユーザが入力中の一時的なカード情報。タイトル、学習内容、タグ、コレクション選択状態を持つ

### Assumptions

- タイトルと学習内容は必須、タグとコレクションは任意とする
- タグはカンマ区切りで複数入力する想定とし、未登録タグ名は新規作成して利用できるものとする
- コレクションは既存候補から選択する想定とし、この画面では新しいコレクションを作成しない
- 利用可能なコレクションがない場合でも、カードはコレクション未設定で登録できる
- 「一覧へ戻る」はカード一覧ページへ戻る導線とする
- 「入力をリセット」は現在画面上で入力中の内容を初期状態へ戻す
- 登録成功後はカード一覧ページへ戻し、成功メッセージで完了を伝える
- 画面イメージに含まれる「下書き保存」は今回の必須スコープには含めず、将来拡張として扱う
- 具体的なユーザー向けメッセージ文言は docs/messages.md に集約し、各画面仕様ではその参照を優先する
- docs/messages.md に集約するのは成功、失敗、バリデーション、補助文言とし、見出し、ボタン名、ラベル、プレースホルダーは画面仕様に残す

### Scope Boundaries

- In Scope: 学習カード登録ページ、ヘッダーメニューへの登録導線追加、カード一覧ページ右上の登録導線追加、タイトル/学習内容/タグ/コレクション入力、入力確認プレビュー、登録・リセット・一覧へ戻る操作、PC/モバイル向け画面構成整理
- Out of Scope: 学習カード編集画面、下書き保存の本実装、復習アルゴリズム、統計集計ロジック、コレクション管理画面そのもの

### Dependencies

- 学習カードを保存できる永続化手段が存在すること
- タグをカードへ紐づける仕組みが存在すること、または追加できること
- コレクション候補を取得して選択肢として表示できること
- カード一覧ページが登録後の戻り先として利用可能であること
- 共通メッセージ定義ファイル docs/messages.md を参照可能であること

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 登録画面表示から登録完了後のカード一覧復帰と成功メッセージ表示までを E2E テストで確認できる
- **SC-002**: ヘッダーメニューとカード一覧右上の両方から登録画面へ遷移できることをフロントエンドテストまたは E2E テストで確認できる
- **SC-003**: タイトルまたは学習内容が不足した場合に不足項目を示すエラーが表示され、入力補完後に再送信できることをフロントエンドテストで確認できる
- **SC-004**: 入力プレビューにタイトル、学習内容、タグ、コレクションの現在値が反映され、入力リセット後に初期状態へ戻ることをフロントエンドテストで確認できる
