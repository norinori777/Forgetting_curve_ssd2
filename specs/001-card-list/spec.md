# Feature Specification: 学習カード一覧（List）

**Feature Branch**: `001-card-list`  
**Created**: 2026-03-07  
**Status**: Draft  
**Input**: User description: "学習カードの一覧画面を提供し、検索・フィルタ・ソート・バルク操作で復習開始へスムーズに遷移できるようにする"

## Clarifications

### Session 2026-03-07

- Q: 「復習開始」は、選択があれば“選択カードのみ”で開始？それとも“現在の絞り込み結果”で開始？ → A: 常に“現在の絞り込み結果”で開始（選択は無視）
- Q: 分割表示は「ページ切替」か「連続読み込み（無限スクロール）」のどちら？ → A: 連続読み込み（無限スクロール）
- Q: 無限スクロールの分割取得はカーソル方式かオフセット方式か？ → A: カーソル方式（`nextCursor`）
- Q: 追加読み込みのトリガは「自動」か「ボタン」か？ → A: 自動（画面下部付近で自動的に追加読み込み）
- Q: 「削除」は復元可能（論理削除）か、復元不可（物理削除）か？ → A: 復元不可（物理削除）。二段階確認はモーダルで行い、モーダル内に削除対象（選択カード）を表示する

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 今日の復習から開始 (Priority: P1)

ユーザは、今日復習すべきカードを一覧から素早く見つけ、復習セッションを開始したい。

**Why this priority**: 学習の継続性に直結する最短ループ（探す→開始）であり、価値が最も高い。

**Independent Test**: 「今日の復習」表示 → 「復習開始」実行 → 表示対象カード群で復習が始まること。

**Acceptance Scenarios**:

1. **Given** ユーザに期限到来（今日以前）のカードが存在する, **When** 一覧で「今日の復習」を表示する, **Then** 期限到来のカードのみが表示される
2. **Given** 「今日の復習」を表示している, **When** 画面上の「復習開始」を実行する, **Then** 表示対象のカード群で復習が開始される（選択状態は影響しない）
3. **Given** 一覧が表示されている, **When** 一覧の任意のカードで「復習開始」を実行する, **Then** 復習フローに遷移できる（開始対象は表示対象に一致する）

---

### User Story 2 - 検索/絞り込みで目的のカードを見つける (Priority: P2)

ユーザは、キーワードやタグ/コレクション等で絞り込み、目的のカードを見つけたい。

**Why this priority**: 学習の準備（探す）のストレスを下げ、復習開始までの摩擦を減らす。

**Independent Test**: キーワード検索とタグ絞り込みを単独で適用し、結果が期待通りに変化すること。

**Acceptance Scenarios**:

1. **Given** タイトルに特定キーワードを含むカードが存在する, **When** キーワード検索を実行する, **Then** 該当カードが一覧に表示される
2. **Given** 特定タグを持つカードが存在する, **When** タグで絞り込む, **Then** 該当タグを持つカードのみが表示される
3. **Given** 一覧が表示されている, **When** ソート条件を「次回復習日時」に変更する, **Then** 表示順が指定通りに入れ替わる

---

### User Story 3 - 複数カードにまとめて操作する (Priority: P3)

ユーザは、複数カードを選択して、アーカイブやタグ付け等の操作を一括で行いたい。

**Why this priority**: 大量カードを扱う際の管理コストを下げ、学習に集中できる。

**Independent Test**: 複数選択 → 一括アーカイブ（またはタグ追加）→ 対象のみ状態が更新されること。

**Acceptance Scenarios**:

1. **Given** 一覧に複数カードが表示されている, **When** 複数カードを選択する, **Then** 選択数が明示されバルク操作が有効になる
2. **Given** 複数カードが選択されている, **When** 「アーカイブ」を実行する, **Then** 選択カードのみがアーカイブされ一覧から取り除かれる（またはアーカイブ表示に切り替わる）
3. **Given** 複数カードが選択されている, **When** 「削除」を実行する, **Then** 削除確認モーダルが表示され削除対象（選択カード）が確認できる。ユーザがモーダルで「削除」を確定した場合のみ削除される（復元不可/物理削除）。キャンセル（またはEsc）した場合は削除されない

---

### Edge Cases

- 検索/フィルタの結果が0件のとき、空状態が分かりやすく表示され、条件を解除できる
- データ取得に失敗したとき、エラーが表示され、ユーザが再試行できる
- 追加読み込み（無限スクロール）の取得に失敗したとき、追加分のみのエラーが分かりやすく表示され、ユーザが再試行できる
- 大量件数でも操作（スクロール/連続読み込み/検索）が破綻せず、ユーザが待たされすぎない
- 権限や状態により操作できないカードが含まれる場合、操作不可が明示され、誤操作にならない
- 日付/時刻がロケール差で誤解を生まない表現（相対/絶対）になっている

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 一覧としてカードを複数件表示できる
- **FR-002**: System MUST 一覧の各項目に「タイトル」「タグ」「次回復習日時」「習熟度」「直近正答率」を表示できる
- **FR-003**: System MUST 検索により表示対象を絞り込める（タイトル/本文/タグ相当の情報を対象）
- **FR-004**: System MUST フィルタにより表示対象を絞り込める（少なくとも「今日の復習」「期限切れ」「未学習」「タグ」「コレクション」に相当）
- **FR-005**: System MUST ソートにより表示順を変更できる（少なくとも「次回復習日時」「習熟度」「作成日」に相当）
- **FR-006**: System MUST 大量件数に対応した分割表示として、連続読み込み（無限スクロール）を提供する（カーソル方式。レスポンスに `nextCursor` を含め、次回取得で `cursor` として利用できる）。ユーザが画面下部付近まで到達したら自動で次のデータを取得し、ロード中であることが分かる表示を行う
- **FR-007**: System MUST 複数選択とバルク操作を提供する（少なくとも「タグ付与/削除」「アーカイブ」「削除」に相当）
- **FR-008**: System MUST 削除操作は二段階確認を必須とし、削除確認はモーダルで行う。モーダルには削除対象（選択カードの件数とタイトルの一覧など）と、削除が復元不可（物理削除）であることを明示する。また、モーダルには「削除を確定する操作」と「キャンセルする操作」を提供し、キャンセル（またはEsc）では削除されない
- **FR-009**: System MUST ローディング/空状態/エラー状態を表示し、ユーザが次の行動（再試行/作成/条件解除）を取れる
- **FR-010**: System MUST 一覧から復習開始へ遷移でき、現在の絞り込み条件（表示対象）に整合したカード群で開始できる（カード選択状態は復習開始対象に影響しない）
- **FR-011**: System MUST キーボード操作のみで主要操作（検索/フィルタ/選択/復習開始/バルク操作）を完了できる（削除確認モーダルの確定/キャンセルも含む）

### Key Entities *(include if feature involves data)*

- **Card**: 学習カード（タイトル、本文、タグ、習熟度、次回復習日時、正答率等の学習状態を持つ）
- **Tag**: カードの分類ラベル（カードは複数タグを持てる）
- **Collection**: カードのまとまり（ユーザが切り替えて一覧を絞り込むための単位）

### Assumptions

- 複数フィルタが同時に指定された場合、原則としてANDで絞り込む（同種の複数指定はOR相当）
- 既定のソートは「次回復習日時が近い順」を優先する
- 日付/時刻はユーザのローカル表記で表示し、相対/絶対の両方を把握できる
- 検索/フィルタ/ソート条件が変わった場合、無限スクロールのカーソルはリセットされ、先頭から再取得される

### Scope Boundaries

- In Scope: 一覧表示、検索、フィルタ、ソート、分割表示（連続読み込み/無限スクロール）、バルク操作、復習開始への遷移、ローディング/空状態/エラー表示
- Out of Scope: カード詳細の編集画面、復習スケジューリング（忘却曲線）アルゴリズム自体の定義

### Dependencies

- 一覧表示に必要なカード属性（次回復習日時、習熟度、正答率等）が、ユーザのカードデータとして取得可能であること
- 復習開始先（復習フロー/セッション）が既に存在し、一覧から対象カード群を渡して開始できること

*Acceptance criteria are primarily defined in the "Acceptance Scenarios" under each User Story.*

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザは一覧画面から復習開始までを 30秒以内に完了できる（一般的なケース）
- **SC-002**: 一覧画面の初回表示で、ユーザが「表示が始まった」と認識できる状態が 2秒以内に得られる（95%の利用で）
- **SC-003**: 検索/フィルタによる目的カード発見の成功率が 90% 以上になる（ユーザテスト/検証で）
- **SC-004**: 一覧画面からの復習開始率（一覧閲覧→復習開始）が改善する（ベースライン比 +10% など）
