# Feature Specification: 学習カード一覧（List）

**Feature Branch**: `001-card-list`  
**Created**: 2026-03-07  
**Status**: Draft  
**Input**: User description: "学習カードの一覧画面を提供し、検索・フィルタ・ソート・バルク操作で復習開始へスムーズに遷移できるようにする"

## Clarifications

### Session 2026-03-07

- Q: 「復習開始」は、選択があれば“選択カードのみ”で開始？それとも“現在の絞り込み結果”で開始？ → A: 常に“現在の絞り込み結果”で開始（選択は無視）
- Q: 分割表示は「ページ切替」か「連続読み込み（無限スクロール）」のどちら？ → A: 連続読み込み（無限スクロール）
- Q: 無限スクロールの分割取得はカーソル方式かオフセット方式か？ → A: カーソル方式（`nextCursor`）
- Q: 追加読み込みのトリガは「自動」か「ボタン」か？ → A: 自動（画面下部付近で自動的に追加読み込み）
- Q: 「削除」は復元可能（論理削除）か、復元不可（物理削除）か？ → A: 復元不可（物理削除）。二段階確認はモーダルで行い、モーダル内に削除対象（選択カード）を表示する

### Session 2026-03-11

- Q: 一覧画面の画面構成はどのUI定義に従う？ → A: `ascii_ui.txt` を画面構成の基準とし、フィルタ配置・一覧構造・各モーダル構成もこれに従う
- Q: UI実装とデザイントークン参照は何に従う？ → A: UI 実装は Tailwind CSS を使用し、色・余白・角丸・フォント・ブレークポイントは `theme.json` を参照する。直書き CSS は避け、既存機能の挙動は変更しない

### Session 2026-03-16

- Q: タグとコレクションの選択UIは別々のモーダルか、単一の共用モーダルか？ → A: 単一の共用モーダルとし、モーダル上部のラジオボタンで「タグ」または「コレクション」を選択できるようにする
- Q: ソート条件は独立した操作行に置くか、一覧表の上部に埋め込むか？ → A: 一覧表の上部に埋め込む
- Q: ステータス系フィルタ（今日の復習・期限切れ・未学習）は個別トグルか、プルダウン選択か？ → A: プルダウンで「今日の復習」「期限切れ」「未学習」から選択する

### Session 2026-03-21

- Q: React Router 対応アプリケーションの共通レイアウトはどの単位で構成する？ → A: ルートに共通レイアウトを置き、ヘッダー・パンくず・フッターを共通化し、各ページは Outlet で切り替える
- Q: 初期表示ルートはどのページにする？ → A: `/` はホームのダミーページを表示し、カード一覧は別ルート `/cards` にする
- Q: パンくずはどの粒度で表示する？ → A: パンくずは常に現在ページ名のみ表示する
- Q: カード一覧以外のリンク先はどのように扱う？ → A: ホーム・復習・統計・設定も実ルートを作り、内容はプレースホルダー表示にする
- Q: バルクの「アーカイブ」実行後、対象カードは一覧上でどう扱う？ → A: アーカイブしたカードは既定の一覧から即時に除外する
- Q: バルクのタグ付与/削除で既存状態と不一致の対象が含まれる場合、どう扱う？ → A: 既に付いているタグの追加や、付いていないタグの削除は no-op として成功扱いにする
- Q: 検索・フィルタ・ソート条件が変わったとき、現在の選択状態はどう扱う？ → A: 条件変更時点で現在の選択をすべてクリアする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 今日の復習から開始 (Priority: P1)

ユーザは、今日復習すべきカードを一覧から素早く見つけ、復習セッションを開始したい。

**Why this priority**: 学習の継続性に直結する最短ループ（探す→開始）であり、価値が最も高い。

**Independent Test**: 「今日の復習」表示 → 「復習開始」実行 → 表示対象カード群で復習が始まること。

**Acceptance Scenarios**:

1. **Given** ユーザに期限到来（今日以前）のカードが存在する, **When** 一覧で「今日の復習」を表示する, **Then** 期限到来のカードのみが表示される
2. **Given** 「今日の復習」を表示している, **When** 画面上の「復習開始」を実行する, **Then** 表示対象のカード群で復習が開始される（選択状態は影響しない）
3. **Given** 一覧が表示されている, **When** 一覧の任意のカードで「復習開始」を実行する, **Then** 復習フローに遷移できる（開始対象は表示対象に一致する）

---

### User Story 2 - 検索/絞り込みで目的のカードを見つける (Priority: P2)

ユーザは、キーワードやタグ/コレクション等で絞り込み、目的のカードを見つけたい。

**Why this priority**: 学習の準備（探す）のストレスを下げ、復習開始までの摩擦を減らす。

**Independent Test**: キーワード検索とタグ絞り込みを単独で適用し、結果が期待通りに変化すること。

**Acceptance Scenarios**:

1. **Given** タイトルに特定キーワードを含むカードが存在する, **When** キーワード検索を実行する, **Then** 該当カードが一覧に表示される
2. **Given** 特定タグを持つカードが存在する, **When** タグで絞り込む, **Then** 該当タグを持つカードのみが表示される
3. **Given** 一覧が表示されている, **When** ソート条件を「次回復習日時」に変更する, **Then** 表示順が指定通りに入れ替わる

---

### User Story 3 - 複数カードにまとめて操作する (Priority: P3)

ユーザは、複数カードを選択して、アーカイブやタグ付け等の操作を一括で行いたい。

**Why this priority**: 大量カードを扱う際の管理コストを下げ、学習に集中できる。

**Independent Test**: 複数選択 → 一括アーカイブ（またはタグ追加）→ 対象のみ状態が更新されること。

**Acceptance Scenarios**:

1. **Given** 一覧に複数カードが表示されている, **When** 複数カードを選択する, **Then** 選択数が明示されバルク操作が有効になる
2. **Given** 複数カードが選択されている, **When** 「アーカイブ」を実行する, **Then** 選択カードのみがアーカイブされ、既定の一覧から即時に除外される
3. **Given** 複数カードが選択されている, **When** 「削除」を実行する, **Then** 削除確認モーダルが表示され削除対象（選択カード）が確認できる。ユーザがモーダルで「削除」を確定した場合のみ削除される（復元不可/物理削除）。キャンセル（またはEsc）した場合は削除されない

---

### Edge Cases

- 検索/フィルタの結果が0件のとき、空状態が分かりやすく表示され、条件を解除できる
- データ取得に失敗したとき、エラーが表示され、ユーザが再試行できる
- 追加読み込み（無限スクロール）の取得に失敗したとき、追加分のみのエラーが分かりやすく表示され、ユーザが再試行できる
- 大量件数でも操作（スクロール/連続読み込み/検索）が破綻せず、ユーザが待たされすぎない
- 権限や状態により操作できないカードが含まれる場合、操作不可が明示され、誤操作にならない
- 日付/時刻がロケール差で誤解を生まない表現（相対/絶対）になっている

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 一覧としてカードを複数件表示できる
- **FR-002**: System MUST 一覧の各項目に「タイトル」「タグ」「次回復習日時」「習熟度」「直近正答率」を表示できる
- **FR-003**: System MUST 検索により表示対象を絞り込める（タイトル/本文/タグ相当の情報を対象）
- **FR-004**: System MUST フィルタにより表示対象を絞り込める（少なくとも、ステータスフィルタとして「今日の復習」「期限切れ」「未学習」をプルダウンから選択でき、加えて「タグ」「コレクション」に相当する絞り込みを提供する）
- **FR-004a**: System MUST タグとコレクションのフィルタUIを入力欄ではなく単一のボタンで提供し、押下時に共用モーダルを表示する
- **FR-004b**: System MUST 共用モーダルの上部にラジオボタンを配置し、「タグ」または「コレクション」を切り替えて選択できるようにする
- **FR-004c**: System MUST 共用モーダル内に検索テキストボックスを配置し、現在選択中の対象（タグまたはコレクション）の候補一覧を絞り込めるようにする
- **FR-004d**: System MUST 共用モーダル内で、現在選択中の対象（タグまたはコレクション）の候補一覧に対してチェックボックスによる複数選択を提供し、適用した選択結果を対応するボタンの右側にカンマ区切りテキストで表示する
- **FR-005**: System MUST ソートにより表示順を変更できる（少なくとも「次回復習日時」「習熟度」「作成日」に相当）
- **FR-005a**: System MUST ソート条件のUIを一覧表の上部に配置し、現在のソート条件と切り替え可能な条件を一覧コンテキスト内で把握できるようにする
- **FR-006**: System MUST 大量件数に対応した分割表示として、連続読み込み（無限スクロール）を提供する（カーソル方式。レスポンスに `nextCursor` を含め、次回取得で `cursor` として利用できる）。ユーザが画面下部付近まで到達したら自動で次のデータを取得し、ロード中であることが分かる表示を行う
- **FR-007**: System MUST 複数選択とバルク操作を提供する（少なくとも「タグ付与/削除」「アーカイブ」「削除」に相当）
- **FR-007a**: System MUST バルクの「アーカイブ」を実行したカードを、成功後ただちに既定の一覧表示対象から除外する
- **FR-007b**: System MUST バルクのタグ付与/削除において、既に付与済みのタグ追加と未付与タグの削除を no-op として成功扱いにし、同一リクエストを繰り返しても結果が不整合にならないようにする
- **FR-007c**: System MUST 検索、フィルタ、またはソート条件が変更された時点で現在のカード選択状態をすべてクリアし、選択数表示とバルク操作の有効状態を新しい表示結果に合わせて更新する
- **FR-008**: System MUST 削除操作は二段階確認を必須とし、削除確認はモーダルで行う。モーダルには削除対象（選択カードの件数とタイトルの一覧など）と、削除が復元不可（物理削除）であることを明示する。また、モーダルには「削除を確定する操作」と「キャンセルする操作」を提供し、キャンセル（またはEsc）では削除されない
- **FR-009**: System MUST ローディング/空状態/エラー状態を表示し、ユーザが次の行動（再試行/作成/条件解除）を取れる
- **FR-010**: System MUST 一覧から復習開始へ遷移でき、現在の絞り込み条件（表示対象）に整合したカード群で開始できる（カード選択状態は復習開始対象に影響しない）
- **FR-011**: System MUST キーボード操作のみで主要操作（検索/フィルタ/選択/復習開始/バルク操作）を完了できる（削除確認モーダルの確定/キャンセルも含む）
- **FR-012**: System MUST 一覧画面の情報配置と主要UI構成を `ascii_ui.txt` に整合させる。少なくとも、検索バー、ステータスフィルタのプルダウン、タグ/コレクション選択ボタン、一覧表上部に埋め込まれたソート、主要アクション、選択バー、カード一覧、ローディング/空状態/エラー状態、削除確認モーダル、タグ/コレクション共用選択モーダルを含む
- **FR-013**: System MUST 一覧画面のUI実装に Tailwind CSS を使用する
- **FR-014**: System MUST 色、余白、角丸、フォント、ブレークポイントを `theme.json` の定義に基づいて適用する
- **FR-015**: System MUST スタイル定義で直書き CSS を避け、`theme.json` に定義されたトークンを優先して利用する
- **FR-016**: System MUST デザイン変更によって既存の検索、フィルタ、ソート、無限スクロール、選択、バルク操作、削除確認、復習開始の挙動を変更しない
- **FR-017**: System MUST React Router を用いた共通レイアウトをルートに配置し、ヘッダー、パンくず、フッターを共通要素として保持したまま、ページ本文は Outlet 配下で切り替える
- **FR-018**: System MUST ヘッダーを固定表示し、ナビゲーションリンク押下時はヘッダー直下のページ表示領域のみを差し替える
- **FR-019**: System MUST 初期表示ルート `/` ではホームページを表示し、カード一覧は `/cards` で表示する
- **FR-020**: System MUST パンくず領域には現在表示中のトップレベルページ名のみを表示する
- **FR-021**: System MUST ヘッダーには「ホーム」「カード一覧」「復習」「統計」「設定」のトップレベルリンクを表示し、各リンクは React Router のルートへ遷移できる
- **FR-022**: System MUST カード一覧以外のトップレベルページについても実ルートを用意し、今回の範囲ではプレースホルダー表示でよい

### Key Entities *(include if feature involves data)*

- **Card**: 学習カード（タイトル、本文、タグ、習熟度、次回復習日時、正答率等の学習状態を持つ）
- **Tag**: カードの分類ラベル（カードは複数タグを持てる）
- **Collection**: カードのまとまり（ユーザが切り替えて一覧を絞り込むための単位）

### Assumptions

- 複数フィルタが同時に指定された場合、原則としてANDで絞り込む（同種の複数指定はOR相当）
- ステータスフィルタ（今日の復習・期限切れ・未学習）はプルダウンによる単一選択とする
- 既定のソートは「次回復習日時が近い順」を優先する
- バルクの「アーカイブ」が成功したカードは、既定の一覧から即時に除外する
- バルクのタグ付与/削除は冪等とし、既存状態と不一致の対象が含まれても no-op として成功扱いにする
- 日付/時刻はユーザのローカル表記で表示し、相対/絶対の両方を把握できる
- 検索/フィルタ/ソート条件が変わった場合、無限スクロールのカーソルはリセットされ、先頭から再取得される
- 検索/フィルタ/ソート条件が変わった場合、現在の選択状態はクリアされる
- `theme.json` は Tailwind CSS のテーマ拡張に利用できる形式を維持する
- デザイン刷新は見た目とスタイリング実装の変更に留め、機能仕様やユーザ操作フローは維持する
- React Router のページ遷移は共通レイアウトを維持し、ヘッダー、パンくず、フッターはページ間で再構成しない
- ホームはダミーページとして提供し、カード一覧のみ既存実装に接続する実ページとする
- 今回の React Router 構成ではトップレベルページのみを扱い、パンくずは階層表現を持たない
- ホーム、復習、統計、設定はプレースホルダーページとして実装し、カード一覧のみ既存の機能画面へ接続する

### Scope Boundaries

- In Scope: 一覧表示、検索、フィルタ、ソート、分割表示（連続読み込み/無限スクロール）、バルク操作、復習開始への遷移、ローディング/空状態/エラー表示、React Router による共通ヘッダー/パンくず/フッター付きレイアウト基盤、カード一覧以外のトップレベルページのプレースホルダールート
- Out of Scope: カード詳細の編集画面、復習スケジューリング（忘却曲線）アルゴリズム自体の定義、カード一覧以外の各ページの本実装内容

### Dependencies

- 一覧表示に必要なカード属性（次回復習日時、習熟度、正答率等）が、ユーザのカードデータとして取得可能であること
- 復習開始先（復習フロー/セッション）が既に存在し、一覧から対象カード群を渡して開始できること
- `ascii_ui.txt` がこの機能の画面構成リファレンスとして維持され、実装と乖離した場合は更新されること
- `theme.json` がこの機能のデザイントークンリファレンスとして維持され、Tailwind CSS から参照可能であること

*Acceptance criteria are primarily defined in the "Acceptance Scenarios" under each User Story.*

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザは一覧画面から復習開始までを 30秒以内に完了できる（一般的なケース）
- **SC-002**: 一覧画面の初回表示で、ユーザが「表示が始まった」と認識できる状態が 2秒以内に得られる（95%の利用で）
- **SC-003**: 検索/フィルタによる目的カード発見の成功率が 90% 以上になる（ユーザテスト/検証で）
- **SC-004**: 一覧画面からの復習開始率（一覧閲覧→復習開始）が改善する（ベースライン比 +10% など）
