# Feature Specification: 設定画面コレクション登録

**Feature Branch**: `001-collection-settings`  
**Created**: 2026-03-28  
**Status**: Draft  
**Input**: User description: "設定画面でコレクションを登録できる画面のデザインをascii uiを作成してください。"

## Clarifications

### Session 2026-03-28

- Q: カードが紐づくコレクションを削除するとき、どう扱うべきか？ → A: そのコレクションにカードが残っている間は削除不可にする

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 設定画面から新しいコレクションを登録する (Priority: P1)

ユーザは、設定画面を開いたまま新しいコレクション名を入力し、学習カードの整理に使えるコレクションを追加したい。

**Why this priority**: コレクションが作成できないと、カード整理や後続の絞り込み体験が成立しないため最優先である。

**Independent Test**: 設定画面のコレクション管理領域で名前を入力して登録し、登録済み一覧に新しいコレクションが追加されることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが設定画面のコレクション管理を開いている, **When** 有効なコレクション名を入力して登録する, **Then** 新しいコレクションが登録済み一覧に追加される
2. **Given** ユーザが登録を完了した, **When** 画面が更新される, **Then** 登録成功が分かる表示とともに新しいコレクションを確認できる
3. **Given** ユーザが登録フォームを見ている, **When** まだ必要条件を満たしていない, **Then** 登録操作を実行しても不完全な入力であることが分かる

---

### User Story 2 - 既存コレクションを変更または削除する (Priority: P2)

ユーザは、既存コレクションの名前や補足情報を見直し、不要なコレクションは安全に削除したい。

**Why this priority**: 登録だけでは運用中の整理ができず、設定画面として不十分なためである。特に削除条件の安全性は誤操作防止に直結する。

**Independent Test**: 既存コレクション一覧から対象を選び、変更内容を保存できることと、カードが残るコレクションでは削除できないことを確認する。

**Acceptance Scenarios**:

1. **Given** すでに複数のコレクションが存在する, **When** ユーザが設定画面を開く, **Then** コレクション名と必要最小限の補足情報を一覧で確認でき、各項目の変更導線と削除導線を識別できる
2. **Given** ユーザが既存コレクションを変更しようとしている, **When** 一覧の「編集」を押す, **Then** 対象コレクションだけを編集できるモーダルが開く
3. **Given** ユーザが編集モーダルを開いている, **When** 有効な内容へ更新して保存する, **Then** モーダルが閉じて一覧に最新内容が反映される
4. **Given** ユーザがカードの残っていないコレクションを削除しようとしている, **When** 削除を確認する, **Then** 対象コレクションだけが一覧から除外される
5. **Given** ユーザがカードの残っているコレクションを削除しようとしている, **When** 削除操作を実行する, **Then** 削除できない理由が表示され、対象は保持される

---

### User Story 3 - 入力失敗時にもやり直せる (Priority: P3)

ユーザは、入力ミスや登録失敗が起きても、内容を見直してその場で再登録したい。

**Why this priority**: 設定作業は短時間で終えたい操作であり、失敗時のやり直しが難しいと離脱につながるためである。

**Independent Test**: 不正な名前や登録失敗を再現し、入力内容の見直しと再実行ができることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが許可されない形式の名前を入力している, **When** 登録しようとする, **Then** どこを直せばよいかがその場で分かる
2. **Given** 一時的な失敗で登録が完了しない, **When** エラーが表示される, **Then** ユーザは入力内容を失わず再試行できる
3. **Given** ユーザが登録を取りやめたい, **When** フォームをリセットまたは閉じる, **Then** 一覧表示へ戻り主導線を見失わない

### Edge Cases

- コレクション名が空欄のまま送信された場合は登録されず、不足内容が明示される
- コレクション名が長すぎる場合でも、入力欄と確認表示の両方で内容が把握できる
- 既存名と完全一致または実質的に同一と判断される名前は、重複登録されない
- 変更時に既存名と同一のまま保存する場合は、不要なエラーにならない
- 編集モーダルを保存せずに閉じた場合でも、一覧の状態は不意に変わらない
- コレクションが0件のときは、空状態から最初の登録へ自然に進める
- カードが残っているコレクションは削除操作を実行しても削除されず、理由が明示される
- 登録処理が失敗しても、ユーザは入力済み内容を保持したまま再試行できる
- 一覧件数が増えても、登録フォームと既存一覧のどちらが主操作か見失わない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 設定画面内にコレクション管理の専用領域を表示する
- **FR-002**: System MUST コレクション管理領域で新しいコレクション名を入力できるようにする
- **FR-003**: System MUST ユーザが入力内容を送信して新しいコレクションを登録できるようにする
- **FR-004**: System MUST 登録済みコレクションを同一画面内で一覧表示し、登録前に確認できるようにする
- **FR-005**: System MUST 一覧の各項目に、ユーザが重複判断に使える最小限の情報を表示する
- **FR-005a**: System MUST 一覧の各項目に、変更操作と削除操作の導線を表示する
- **FR-006**: System MUST コレクション名の必須条件を満たさない場合、登録前に分かる形で入力エラーを示す
- **FR-007**: System MUST 既存コレクションと重複する名前を登録しようとした場合、重複であることを示して登録を完了させない
- **FR-008**: System MUST 登録成功時に、成功したことが分かる表示と最新の一覧状態を返す
- **FR-009**: System MUST 登録失敗時に、ユーザが原因を認識し入力内容を保持したまま再試行できるようにする
- **FR-010**: System MUST コレクションが存在しない場合、空状態と最初の登録導線を表示する
- **FR-011**: System MUST 登録フォームに、少なくとも「登録する」と「入力を取り消す」または同等の操作を提供する
- **FR-011a**: System MUST 既存コレクションを変更できるようにし、更新成功時は一覧に最新状態を反映する
- **FR-011aa**: System MUST 既存コレクションの変更操作は一覧の各行から開始し、対象だけを編集するモーダルで完結できるようにする
- **FR-011ab**: System MUST 編集モーダルには、少なくとも保存操作とキャンセル操作を提供し、キャンセル時は未保存の変更を一覧へ反映しない
- **FR-011b**: System MUST 既存コレクションの削除前に、対象を誤認しない確認段階を提供する
- **FR-011c**: System MUST カードが1枚以上紐づくコレクションは削除不可とし、削除できない理由を画面上で明示する
- **FR-012**: System MUST PC表示とモバイル表示の両方で、既存一覧の確認と新規登録の主導線を維持する
- **FR-013**: System MUST 設定画面全体の中で、コレクション管理が他の設定項目と区別できる見出しと説明を持つ
- **FR-014**: System MUST 画面の情報配置と状態変化を ascii_ui.txt の画面リファレンスと整合させる

### Key Entities *(include if feature involves data)*

- **Collection**: 学習カードを束ねるユーザ管理単位。名前と一覧表示用の補足情報を持つ
- **Collection Draft**: ユーザが設定画面で入力中の新規コレクション情報
- **Collection Edit Draft**: ユーザが既存コレクションの変更中に保持する一時的な編集情報
- **Collection Edit Modal**: 一覧から選択した単一コレクションを安全に更新するためのモーダル表示単位
- **Collection Settings View**: 設定画面内で、既存一覧、登録フォーム、成功状態、エラー状態をまとめて扱う表示単位

### Assumptions

- 設定画面には複数の設定カテゴリが存在し、その1つとしてコレクション管理を表示する
- コレクション登録で最低限必要な入力は名前のみとする
- 既存コレクション一覧には、少なくとも名称と識別の助けになる補足情報を表示する
- 変更時も、最低限編集できる項目は名前と既存一覧で識別に使う補足情報とする
- 既存コレクションの編集は別ページ遷移ではなく、一覧文脈を保ったモーダル内で行う
- 失敗時の再試行では、ユーザが入力済みの名称を再入力しなくてよい状態を維持する
- カードが紐づくコレクションは削除不可とし、カードの所属先をこの画面で暗黙変更しない
- 空状態、成功状態、入力エラー状態、登録失敗状態をこの画面の必須状態として扱う
- 詳細な配色や実装技術ではなく、情報構造と操作導線を今回の仕様対象とする

### Scope Boundaries

- In Scope: 設定画面におけるコレクション管理の情報設計、登録フォーム、既存一覧、変更操作、削除確認、削除不可状態、空状態、入力エラー、登録失敗、成功表示、PC/モバイル向け ASCII UI リファレンス
- Out of Scope: コレクションの並び替え、権限管理、カードへの一括再割当、削除時の自動再配置、実装技術の詳細設計

### Dependencies

- ユーザが既存コレクションを取得して確認できること
- 新しいコレクション登録結果を一覧へ反映できること
- 既存コレクションの更新結果と削除可否を一覧へ反映できること
- 設定画面全体のナビゲーションと共通レイアウトが別途存在すること

### Screen Reference

- 画面構成の詳細は ascii_ui.txt を基準とする
- ASCII UI には、標準状態、編集モーダル、削除確認、削除不可、空状態、入力エラー、登録失敗、モバイル表示を含める

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: ユーザは設定画面を開いてから 1分以内に新しいコレクションを登録完了できる
- **SC-002**: 初めて利用するユーザでも、主要操作を迷わず見つけて登録まで到達できる割合が 90% 以上になる
- **SC-003**: 重複名の登録試行時に、ユーザが修正方法を理解できる割合が 90% 以上になる
- **SC-004**: コレクション未作成の状態でも、最初の1件目を登録できる完了率が 95% 以上になる
- **SC-005**: 既存コレクションの変更または削除を試みたユーザが、成功または削除不可理由を迷わず理解できる割合が 90% 以上になる
