## User Scenarios & Testing *(mandatory)*

### User Story 1 - 本文をより広く表示する (Priority: P1)

ユーザは、どの画面でもヘッダーに場所を取られすぎず、本文や主要操作をより早く見たい。

**Why this priority**: 学習開始や画面内操作の前に見える情報量が少ないと、主要タスクに入るまでの摩擦が増えるためである。

**Independent Test**: 任意のトップレベルページを開き、パンくず専用領域がなく、従来より小さいヘッダーの下に本文がより多く見えることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが任意のトップレベルページを開く, **When** 画面の先頭領域を確認する, **Then** パンくず専用領域は表示されず、ヘッダーは従来よりコンパクトに表示される
2. **Given** ユーザが主要画面を開く, **When** 初回表示を見る, **Then** ヘッダー直下にページ本文または主要操作がより多く表示される

---

### User Story 2 - パンくずがなくても迷わず移動する (Priority: P2)

ユーザは、パンくずが表示されなくても、今いるページと移動先を迷わず把握したい。

**Why this priority**: 高さを削減できても、現在地や画面間移動が分かりにくくなると、全体の使いやすさが下がるためである。

**Independent Test**: 複数のトップレベルページ間を移動し、パンくずがなくても現在地と移動先を識別できることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが任意のトップレベルページにいる, **When** ヘッダーメニューを見る, **Then** ホーム以外の既存トップレベル移動先を引き続き選択できる
2. **Given** ユーザがトップレベルページを切り替える, **When** 切り替え後の画面を見る, **Then** パンくずがなくても現在表示中のページ名または同等の現在地情報を認識できる
3. **Given** ユーザがホーム以外のページにいる, **When** ヘッダーのアイコンまたはサービス名を選択する, **Then** ホームへ遷移できる

---

### User Story 3 - 小さい画面でも圧迫感を減らす (Priority: P3)

ユーザは、モバイル幅を含む小さい画面でも、共通ヘッダーに縦方向の表示領域を奪われすぎずに使いたい。

**Why this priority**: 画面が狭いほど固定ヘッダーの高さによる影響が大きく、操作開始までに余分なスクロールが発生しやすいためである。

**Independent Test**: 狭い表示幅でもトップレベルメニューへアクセスでき、初回表示で本文の見える範囲が改善していることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが小さい画面幅で任意のトップレベルページを開く, **When** 初回表示を見る, **Then** 共通ヘッダーが必要以上に縦方向を占有せず、本文に到達しやすい
2. **Given** ユーザが小さい画面幅で操作している, **When** 他のトップレベルページへ移動したい, **Then** パンくずがなくてもトップレベルメニューとブランド領域経由で移動できる

### Edge Cases

- ページ名やナビゲーション項目が長い場合でも、ヘッダーが再び大きな空き領域を生まない
- 直接URLでページを開いた場合でも、パンくずなしで現在地が分かる
- ホームのように階層表現が不要なページでも、見出しや導線が不足しない
- 固定ヘッダーの高さ変更によって、本文先頭が隠れたり操作要素と重なったりしない
- ヘッダーのアイコンまたはサービス名が長い表示幅でも、ホーム遷移のクリック領域が失われない

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 共通レイアウトからパンくず専用表示領域を削除する
- **FR-002**: System MUST パンくず領域の削除に合わせて、共通ヘッダーの縦方向の占有量を現状より小さくする
- **FR-003**: System MUST 共通ヘッダー内のアイコンとサービス名をホームへの導線として維持し、どのトップレベルページからでもホームへ戻れるようにする
- **FR-004**: System MUST パンくずを削除した後も、ユーザが現在表示中のページを画面内で認識できるようにする
- **FR-005**: System MUST ページ遷移時に共通ヘッダーを維持しつつ、本文領域のみが切り替わる構成を保つ
- **FR-006**: System MUST デスクトップ幅とモバイル幅の双方で、ヘッダー簡素化後もホーム以外のトップレベルメニューとブランド領域によるホーム遷移を利用できるようにする
- **FR-007**: System MUST ヘッダーメニューからホームリンクを削除し、ホームへの遷移はアイコンまたはサービス名の選択に統一する
- **FR-008**: System MUST ヘッダーの高さ変更後も、本文先頭の情報や主要操作が固定ヘッダーに隠れないようにする
- **FR-009**: System MUST パンくずに依存していた現在地の認識を、各ページで見えるページ名または同等の文脈情報で補えるようにする
- **FR-010**: System MUST この変更を、共通レイアウトを利用する全対象ページに一貫して適用する
- **FR-011**: System MUST ヘッダーの見た目を簡素化しても、ホーム導線の表現変更を除き、既存ページの主機能や遷移先の動作を変更しない

### Assumptions

- 共通レイアウトを利用する各トップレベルページは、本文側でページ名または現在地を示す見出しを表示できる
- トップレベルページ自体は増減せず、ホームだけはメニュー項目ではなくブランド領域から遷移する
- フッターの有無や内容は今回の変更対象外とする
- 変更対象は共通ヘッダーとその直下の共通導線領域であり、各ページ固有の業務機能は維持する

### Scope Boundaries

- In Scope: 共通ヘッダーの縦幅縮小、パンくず専用領域の削除、現在地認識の維持、デスクトップとモバイルでの共通レイアウト調整
- Out of Scope: 新しいナビゲーション項目の追加、ページ情報構造の再設計、各ページ固有機能の追加変更、フッター仕様の変更

### Dependencies

- 現在の共通レイアウトが、全対象ページで同じヘッダー構成を共有していること
- 各トップレベルページが、パンくずの代わりに現在地を示せる見出しまたは文脈情報を持てること
- ヘッダーのアイコンとサービス名が、全対象画面で一貫してホーム導線として扱えること
- ホーム以外の既存トップレベルメニュー遷移が、ヘッダー簡素化後も継続利用できること

*Acceptance criteria are primarily defined in the "Acceptance Scenarios" under each User Story.*

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 代表的なトップレベルページにおいて、初回表示時にパンくず専用領域が 100% 表示されない
- **SC-002**: デスクトップ幅とモバイル幅の双方で、初回表示時に本文または主要操作として視認できる情報量が現状より増える
- **SC-003**: 利用確認において、90% 以上のユーザがパンくずなしでも現在地と別のトップレベル移動先を初回試行で識別できる
- **SC-004**: 受け入れ確認の対象となる全トップレベルページで、ブランド領域からホームへ戻る遷移とホーム以外の主要ナビゲーション遷移成功率が 100% 維持される
