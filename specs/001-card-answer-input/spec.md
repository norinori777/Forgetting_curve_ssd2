# Feature Specification: 学習カード登録の回答入力項目

**Feature Branch**: `001-card-answer-input`  
**Created**: 2026-03-22  
**Status**: Draft  
**Input**: User description: "学習カードの登録に回答を入力項目を追加。回答は任意項目となる。複数行入力可能の入力項目とする。"

## Clarifications

### Session 2026-03-22

- Q: 回答が未入力または空白のみの場合はどう扱う？ → A: 回答が未入力または空白のみなら未登録として保存する
- Q: 登録前プレビューでは回答をどこまで表示する？ → A: プレビューでは回答を全文表示する

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 回答付きで学習カードを登録する (Priority: P1)

ユーザは、問題や学習内容だけでなく、その回答も一緒に記録した状態で新しい学習カードを登録したい。

**Why this priority**: 回答を入力できなければ、この feature の主目的である「登録時に回答を持たせること」が成立しないため。

**Independent Test**: 学習カード登録画面でタイトル、学習内容、回答を入力して登録し、保存されたカードに回答が含まれていることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが学習カード登録画面を開いている, **When** タイトル、学習内容、回答を入力して登録する, **Then** 新しい学習カードが回答付きで登録される
2. **Given** ユーザが回答欄に複数行の内容を入力している, **When** 登録を完了する, **Then** 入力した改行を保った回答内容がカードに含まれる
3. **Given** ユーザが回答を入力済みで登録前の確認をしている, **When** 画面内の入力内容を見直す, **Then** 回答欄に入力された内容を他の項目と同様に確認できる
4. **Given** ユーザが複数行の回答を入力済みである, **When** 登録前プレビューを確認する, **Then** 回答は省略されず全文を確認できる

---

### User Story 2 - 回答なしでも学習カードを登録する (Priority: P1)

ユーザは、回答が不要な学習カードについては回答を空欄のまま登録したい。

**Why this priority**: 回答を任意項目として扱うことが、この feature の前提条件だから。

**Independent Test**: 学習カード登録画面で回答欄を空欄のまま登録し、カード登録自体は完了することを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが学習カード登録画面を開いている, **When** タイトルと学習内容だけを入力して登録する, **Then** 回答未入力でもカード登録は完了する
2. **Given** ユーザが回答欄を空欄のままにしている, **When** 登録操作を実行する, **Then** 回答未入力だけを理由に登録が拒否されない
3. **Given** ユーザが回答欄を未入力のまま他の必須項目を満たしている, **When** 登録完了後のカードを確認する, **Then** 回答が未登録であることだけが反映され、他の登録内容は保持される

---

### User Story 3 - 長い回答を無理なく入力する (Priority: P2)

ユーザは、説明文や手順のような長めの回答でも、途中で切れずに複数行で入力したい。

**Why this priority**: 回答が単語だけではなく説明文になるケースでは、複数行入力できないと実用性が不足するため。

**Independent Test**: 回答欄に 3 行以上の内容を入力し、入力中の視認性を保ったまま登録まで完了できることを確認する。

**Acceptance Scenarios**:

1. **Given** ユーザが長文回答を入力したい, **When** 回答欄に複数行のテキストを入力する, **Then** 入力欄は複数行入力に対応し、改行を含めて入力できる
2. **Given** ユーザが回答欄に複数行を入力している, **When** 入力途中で内容を見直す, **Then** 各行の内容を視認できる
3. **Given** ユーザが複数行回答を入力済みである, **When** 登録処理が完了する, **Then** 回答内容は 1 行に潰されず登録内容として扱われる

### Edge Cases

- 回答欄が空欄でも、タイトルと学習内容が満たされていれば登録できること
- 回答欄に先頭や末尾の空白だけが入力されている場合は未登録として扱えること
- 回答欄に先頭や末尾の空行を含む場合でも、空白のみではない入力はユーザが意図した複数行入力として扱えること
- 非常に短い回答と非常に長い回答のどちらでも入力できること
- 回答欄だけ未入力、または回答欄だけ入力済みでも、必須項目の判定はタイトルと学習内容に対してのみ行われること
- 登録失敗時に、回答欄へ入力した複数行内容が失われず再試行できること

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST 学習カード登録画面に回答入力項目を追加する
- **FR-002**: System MUST 回答入力項目を任意項目として扱う
- **FR-003**: System MUST タイトルと学習内容の必須判定に、回答未入力を含めない
- **FR-004**: System MUST 回答が未入力でも学習カード登録を完了できるようにする
- **FR-004a**: System MUST 回答が未入力または空白のみの場合、未登録として扱う
- **FR-005**: System MUST 回答が入力された場合、その内容を学習カードの登録内容に含める
- **FR-006**: System MUST 回答入力項目で複数行テキストを入力できるようにする
- **FR-007**: System MUST 回答入力項目で入力した改行を保持したまま登録内容として扱う
- **FR-008**: System MUST 登録前にユーザが入力内容を確認できる場合、回答入力内容も他の登録項目と同様に確認対象へ含める
- **FR-008a**: System MUST 登録前プレビューでは、回答入力内容を省略せず全文表示する
- **FR-009**: System MUST 登録に失敗した場合、回答入力項目の内容を失わずに再試行できるようにする
- **FR-010**: System MUST 回答入力項目を追加しても、既存のタイトル、学習内容、タグ、コレクション登録の流れを継続利用できるようにする

### Key Entities *(include if feature involves data)*

- **Learning Card**: タイトル、学習内容、任意の回答、タグ、コレクションを持つ学習対象
- **Answer Input**: 学習カード登録時に入力される任意の回答内容。複数行テキストを許可する
- **Learning Card Draft**: 登録画面上で一時的に保持される入力内容。タイトル、学習内容、回答、タグ、コレクションの状態を持つ

### Assumptions

- 既存の学習カード登録画面と登録導線はすでに存在する
- タイトルと学習内容は引き続き必須項目とする
- 回答は保存時点では 1 カードにつき 1 つの入力項目として扱う
- 回答は装飾付きテキストではなく、複数行の通常テキストとして扱う
- 回答が未入力または空白のみの場合は、未登録として保存する
- 登録失敗時は既存の再試行体験に回答入力項目も含めて維持する

### Scope Boundaries

- In Scope: 学習カード登録画面への回答入力項目追加、任意入力としての扱い、複数行入力、登録内容への反映、登録失敗時の入力保持
- Out of Scope: 回答の表示ルール変更、回答の検索対象追加、既存カード編集画面への回答入力追加、回答を使った採点ロジック変更

### Dependencies

- 既存の学習カード登録画面が利用可能であること
- 学習カード登録内容を保存する仕組みが存在すること
- 既存の登録成功・失敗処理に入力内容保持の仕組みが存在するか、同等の扱いを追加できること

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 回答ありの登録ケースが受け入れ確認対象の 100% で正常に完了する
- **SC-002**: 回答なしの登録ケースが受け入れ確認対象の 100% で正常に完了する
- **SC-003**: 3 行以上の回答入力ケースが受け入れ確認対象の 100% で改行を保って登録できる
- **SC-003a**: 回答プレビュー確認ケースが受け入れ確認対象の 100% で全文表示される
- **SC-004**: 登録失敗を伴う確認ケースで、回答欄の入力内容保持が受け入れ確認対象の 100% で維持される
- **SC-005**: 受け入れ確認参加者の 90% 以上が、回答欄が任意項目であることを画面上で正しく理解できる
