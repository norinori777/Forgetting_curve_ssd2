# 憲法のテンプレート

## 目的と成功条件: このリポジトリ/プロダクトで「成功」を何で測るか（例: 学習完了率、速度、正確性）

- 目的: 忘却曲線（間隔反復）に基づき、ユーザーが「最小の学習時間で最大の記憶保持」を達成できる暗記支援アプリを提供する
- 目的: 復習タイミングの意思決定コストを下げ（いつ復習すべきか迷わない）、学習の継続を阻害する摩擦を減らす

- 成功条件（推奨の測定例。数値はプロダクト段階に合わせて調整）

  - 記憶保持: 予測された復習タイミングでの想起成功率（正答率）が、一定期間（例: 7日/30日）で改善または維持される
  - 継続性: 週次アクティブ（WAU）や復習セッション完了率が一定水準を満たし、学習が「続く」状態を作れる
  - 効率: 1セッションあたりの復習完了数、または1カードあたりの平均復習時間が悪化せず、学習負荷が過度に増えない
  - 体験: 重要タスク（追加→復習開始→記録）までの手順数/時間が小さく、オンボーディングで迷いが少ない
  - 信頼性: クラッシュやデータ消失が実質ゼロ（例: crash-free率、バックアップ成功率）で、学習ログが守られる
  - 説明可能性: スケジューリングの根拠（なぜ今日復習か）が簡潔に説明でき、ユーザーが納得して使える
  - プライバシー: 学習データは最小限収集・明示された目的でのみ利用し、ユーザーがエクスポート/削除できる

  ## 言語とコミュニケーション（ローカライズ）

  - リポジトリ内の仕様書・憲法・PR/Issue テンプレート、開発ドキュメントは日本語を一次言語とする。
  - GitHub 上での作業中のコメントやレビューコメントも日本語で行うことを原則とする（チーム内の合意がある場合は例外を許容）。
  - 国際的な共同作業や外部向け公開が必要な場合は、簡潔な英語サマリを付ける（英訳は簡潔に、完全翻訳は必要時に実施）。
  - 自動生成メッセージや CI 出力は可能な範囲で日本語化し、英語併記を推奨する。重要なエラーメッセージは英日両方で記載すること。
  - 用語集を用意して用語の一貫性を保つ（例: "忘却曲線" = "spaced repetition"）。
  - PR や Issue のテンプレートも日本語版を用意し、必要に応じて英語版を併設する。

  ## スコープ境界: やること/やらないこと、非目標（機能追加の暴走を止める）

- やること（このプロジェクトが優先して取り組む機能・範囲）

  - 基本的なカード（フラッシュカード）作成・編集・タグ付け・インポート/エクスポート機能
  - 忘却曲線（間隔反復）に基づく自動スケジューリングと通知（オプトイン）
  - 各カードの想起履歴・正答率・次回推奨復習日時の記録と表示
  - シンプルで素早く復習できるUI（追加→復習開始→記録が3タップ以内を目標）
  - 最小限のオフライン対応（ローカル保存と同期失敗時のキュー）
  - ユーザーデータのエクスポート／完全削除機能、プライバシーに関する明示的設定

- やらないこと（現段階では意図的に除外することで開発を加速）
  - 大規模なソーシャル機能（フォロー、公開デッキのランキング、コメントなど）
  - 高度なコンテンツ生成（AIによる自動カード生成や大量のマルチメディア編集）※将来的な拡張は可
  - 複雑なゲーム化（ガチャ、仮想通貨、複雑な報酬経済）
  - 企業向け大規模分析プラットフォーム（SaaSの多ユーザー運用や企業データ統合）

- 非目標（誤解されやすいが本プロジェクトで追わない価値観や制約）

  - 「万能の最適スケジューラ」を研究目的に突き詰めること（アカデミックな最適化は別プロジェクト）
  - 全ての学習領域に特化したカスタムアルゴリズムの提供（まずは汎用的な間隔反復モデルで十分）
  - すべてのプラットフォームでの同時対応（MVPはまずWebまたは1つのモバイルプラットフォームを優先）

- 補足ルール（スコープ運用のための実務的ガイドライン）
  - MVP基準: コアループ（カード追加→定期復習→履歴確認）が完結していること。
  - 新機能は「継続性」「効率」「信頼性」のいずれかに明確に貢献する場合のみ追加を検討。
  - 依存追加は軽量ライブラリ優先。大きな外部サービス導入は設計レビュー必須。
  - 将来的に除外した機能を再検討する際は、必ずADRに理由と条件を残す。

## 原則（優先順位つき）: 例「正確性 > 速度」「可読性 > マイクロ最適化」など、衝突時の裁定ルール

- 優先順位付き原則（上が高い優先度）

  1. 正確性（Accuracy）: 復習スケジューリングと想起評価の正確性を最優先とする。誤った推奨は学習成果を損なうため、ある程度の計算コストや説明負担は許容する。
  2. 継続性・UX（Continuity & UX）: ユーザーが継続できる体験を重視する。主要フローの摩擦を最小化し、オンボーディングの迷いを減らす。
  3. プライバシーとデータ最小化（Privacy & Minimal Data）: 必要最小限のデータのみ収集し、ローカルファースト設計を優先。ユーザーがデータをエクスポート・削除できることを保証する。
  4. 説明可能性（Explainability）: スケジューリングの根拠がユーザーに説明できること。なぜ今日復習が推奨されるのかを簡潔に示すUI/文言を提供する。
  5. 信頼性（Reliability）: データの整合性とクラッシュ回避を重視する。バックアップ・同期失敗の取り扱いを明確にする。
  6. セキュリティ（Security）: ユーザーデータ保護（暗号化や認可）を確実に行う。
  7. パフォーマンス（Performance）: 応答性はUXの観点で十分確保する。正確性とのトレードオフが発生する場合は原則的に正確性優先だが、ユーザー体験を損なう閾値を超える場合は性能改善を行う。
  8. 保守性・可読性（Maintainability & Readability）: 将来の改修を容易にするため、コード可読性とテストを重視する。
  9. 依存最小化（Minimal Dependencies）: 軽量な依存を好む。大きな外部サービス導入は設計レビューを必須とする。
  10. 拡張性（Extensibility）: 将来的な機能追加（例: AI支援、共有機能）を見据え、API互換性を意識した設計を行う。

- 衝突時の裁定ルール（実務ルール）
  - 正確性 vs 速度: 基本は正確性優先。だがレスポンス遅延が主要フロー（復習体験）に影響する場合は性能改善を行い、正確性低下を最小化する代替案を提示する。
  - プライバシー vs 分析: プライバシーを優先。分析に必要なデータは匿名化・集約で対応し、個人識別情報はデフォルトで収集しない。
  - UX（継続性） vs 高度機能: 継続性を損なわない範囲でのみ新機能を導入。複雑化が懸念される場合はオプトイン機能として後回しにする。

- 運用ルール
  - 設計判断はADR（設計決定記録）で残す。重要なトレードオフは必ず理由と代替案を記載する。
  - 重大な例外扱いはチーム合意の上で文書化する。
  - 具体的な閾値（例: 許容レスポンス時間、クラッシュ率基準）はプロダクト段階で別途 KPI にて定義する。

## 品質基準（Definition of Done）: テスト、型/静的解析、ドキュメント更新、エラー処理、ログ方針

- 必須の DoD チェック（PR マージ前に満たすこと）
  - テスト: ユニットテストでコアロジック（スケジューラ、復習判定、保存/同期）をカバー。新機能は少なくとも1つのテストを追加。統合テストは同期／バックアップ周りを含む。E2Eは主要コアループ（追加→復習→履歴）が動くことを確認。
  - 型・静的解析: TypeScriptなら `tsc`/`eslint`、Javaなら `mvn`/`gradle` プラグイン（Checkstyle、SpotBugs、PMD、ErrorProne 等）や `spotless` 等を用いて静的チェックがパスすること。CI で自動実行し、警告やエラーは修正する。
  - ドキュメント: ユーザー向け（オンボーディング、プライバシーポリシー、エクスポート/削除手順）および開発者向け（API/データモデル、マイグレーション手順）を更新。ユーザーに影響する変更は `CHANGELOG.md` に記載。
  - エラー処理: 入力検証を行い、ユーザーに分かりやすいエラーメッセージを提示する。ネットワーク同期はリトライ/バックオフとローカル保存のフォールバックを実装。重大な例外は可観測にして無音失敗を避ける。
  - ログ方針: 構造化ログ（JSON等）を用い、`INFO/ERROR/DEBUG` レベルを厳格に使う。個人情報はログに残さない（PII レッドアクション）。セッションIDやリクエストIDで相関可能にする。

- CI・運用ルール
  - CI ガード: テスト（unit+integration+E2E）、型チェック、静的解析、セキュリティスキャンがすべて通過しない限りマージ不可。
  - コードレビュー: 少なくとも1人のレビュアーが承認すること。アルゴリズムや設計に重大な変更がある場合は2名以上を推奨。
  - リリース準備: ユーザー向け変更は `CHANGELOG.md` を更新し、データマイグレーションがある場合はローカル/ステージングでの回帰検証を必須とする。

- データ整合性・バックアップ
  - マイグレーションはダウン/アップ両方向テストを用意し、デプロイ前にバックアップを取得できること。
  - データ消失を防ぐための自動バックアップ／復旧手順を文書化する。

- 観測性とアラート
  - キー指標（WAU、復習完了率、想起成功率、クラッシュ率）を計測し、閾値を超えたらアラートを設定する。
  - 重大インシデント発生時のエスカレーション手順を文書化。

- セキュリティ・プライバシー要件
  - 保存データの暗号化（必要に応じて at-rest）、通信の TLS 強制、認可/認証の基本要件を満たすこと。
  - ログ・メトリクスに PII を含めない。データ削除・エクスポート機能をユーザーが利用可能にする。

- 追加の運用ガイドライン
  - 変更が DoD に合致するか疑義がある場合は ADR を作成し、チームで合意した上で例外を記録する。
  - 重要なトレードオフ（例: 精度とレスポンス）については、代替案と影響範囲を PR 説明に明記する。

## 変更の進め方: PRサイズ、レビュー要件、ブランチ戦略、マージ方式、リリース/バージョニング

- PR サイズと内容のガイドライン
  - 小さく、単一目的: 1 PR = 1 意図（バグ修正 / 小機能 / ドキュメント / テスト）を原則とする。可能な限り 200 行未満を目安に分割する。
  - 大規模変更は RFC/提案イシューを先に作成し、設計合意後に複数 PR に分割する。

- レビュー要件
  - あらゆるコード変更は最低1名の承認を必須とする。設計・アルゴリズム変更や依存追加は2名以上の承認を推奨。
  - セキュリティ／プライバシーに関わる変更はセキュリティ担当者の確認を受ける。
  - レビューチェックリスト: DoD の満たし具合（テスト・静的解析・ドキュメント・マイグレーション手順）を確認すること。

- ブランチ戦略
  - 保護ブランチ: `main` は直接 push 禁止、保護ルール（CI 必須、レビュー必須）を適用する。
  - 機能開発は `feature/<短い説明>`、バグは `fix/<issue-number>-<短い説明>`、リリースは `release/vX.Y.Z` を使う。
  - 長期ブランチ（`develop` 等）は不要だが、チームの運用に合わせて採用可。重要なのは一貫性。

- マージ方式
  - マージは原則 `squash and merge` を推奨（履歴の簡潔化）。履歴を残したい場合は `merge commit` を許可する運用ルールを追加する。
  - リベースは PR 作成者が必要に応じて行い、マージ前にコンフリクトを解消する。

- リリースとバージョニング
  - バージョニングは SemVer（`MAJOR.MINOR.PATCH`）を採用。Breaking change は MAJOR を上げる。
  - リリースプロセス: `release` ブランチ作成 → 回帰テスト・QA → タグ付け（`vX.Y.Z`）→ CHANGELOG 更新 → 本番デプロイ。
  - ホットフィックスは `hotfix/vX.Y.Z` ブランチを切り、修正後 `main` と該当 release にマージする。

- CI/CD とマージガード
  - CI がパス（テスト・静的解析・セキュリティスキャン）しない限りマージ不可。
  - 自動リリース／デプロイはステージング環境での承認を必須とする。

- データ・マイグレーションポリシー
  - DB/データモデルの変更は必ずマイグレーション手順を同梱し、ロールアウト段階でのフォールバック計画を用意する。
  - 破壊的変更は複数段階（互換レイヤー→移行→削除）で実施する。

- CHANGELOG とリリースノート
  - すべてのユーザー向け変更は `CHANGELOG.md` に記載する。テンプレートに PR 番号・概要・影響範囲を明記。

- その他運用ルール
  - 新しい外部依存を追加する際はセキュリティ評価とライセンス確認を必須とする。
  - 大きな仕様変更は事前に ADR/RFC を作成し、合意を得る。
  - 緊急パッチは `hotfix` 流儀を使い、事後に手順と原因を文書化する。
