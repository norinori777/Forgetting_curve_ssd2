# 忘却曲線アプリ — 一覧（List）仕様

## 概要

- ステータス: Draft
- 目的: 学習カード（項目）の一覧を表示し、検索・絞り込み・並び替え・一括操作を提供して復習ワークフローへスムーズに遷移させる。
- 対象ユーザ: 日常的に学習カードを作成・復習する学習者（モバイル／デスクトップ）。

## ゴール

- 必要なカードを素早く見つけ、復習を開始できること。
- 一覧上で学習状況（次回復習日時、マスタリーレベル等）が一目で把握できること。
- フィルタやソートで学習優先度に従ったワークフローを実現すること。

## 利用シナリオ（ユーザーストーリー）

- ユーザは今日復習すべきカードだけを一覧で見たい。
- ユーザはタグやコレクションでカードを絞り込み、まとめて復習を開始したい。
- ユーザは個別カードを編集／削除／アーカイブできることを期待する。

## 要求事項

### 機能要件（必須）

- FR-1: 一覧表示 — カードを行（またはカードUI）で表示する。各行は以下の情報を表示する: タイトル、タグ、次回復習日時、マスタリーレベル（数値やバッジ）、直近の正答率（%）、アクションボタン（復習開始、編集、削除）。
- FR-2: 検索 — タイトル・本文・タグでの全文検索を提供する。インクリメンタル検索可。
- FR-3: フィルタ — 以下のプリセットフィルタを提供: 「今日の復習」「期限切れ」「未学習」「タグでフィルタ」「コレクションでフィルタ」。複数フィルタのAND/OR組合せをサポートする（優先度は設計で決定）。
- FR-4: ソート — 次回復習日時（昇順/降順）、マスタリーレベル、作成日でソート可能。
- FR-5: ページネーション／無限スクロール — 大量データ対応。既定はページネーションだが設定で切替可。
- FR-6: バルク操作 — 選択した複数カードに対して「復習開始」「タグ付与/削除」「アーカイブ」「削除」を行える。確認ダイアログ必須（削除は二段階確認）。
- FR-7: 状態表示 — ローディング、空一覧（ヒントとCTA）、エラー表示（再試行ボタン）を用意。
- FR-8: 移動/開始 — 「復習開始」を押すと復習セッションが適切な項目群で開始される（選択中のフィルタ/ソートを反映）。

### 非機能要件

- NFR-1: 応答性 — 一覧表示は常に2秒以内に初期応答（ローディング含む）。ページ切替はUX上速やかに。
- NFR-2: セキュリティ — 削除・編集は認可チェックを行う。APIは認証済みトークンを要求。
- NFR-3: スケーラビリティ — 10万件クラスのカード一覧を想定したページネーション／クエリ最適化。
- NFR-4: 国際化 — 日本語を含む多言語に対応。日付/時間はローカル形式で表示。

## UI/UX 詳細

- レイアウト:
  - ヘッダ: ページタイトル、グローバル検索、新規作成ボタン、インポート/エクスポートボタン。
  - サイドバー（任意）: タグ一覧・コレクションリスト・プリセットフィルタ。
  - メイン: カード行のリスト（テーブル形式またはカードグリッド）。各行の左端に選択チェックボックス（バルク操作用）。
- 行要素（優先順）:
  1. 選択チェックボックス
  2. タイトル（クリックで詳細/編集）
  3. タグ（省略時は「…」で展開）
  4. 次回復習日時（相対時刻と絶対時刻をホバーで切替）
  5. マスタリーレベル（バッジ or 星）
  6. 直近正答率（小さなチャート or %）
  7. アクションボタン（復習（primary）、編集（icon）、削除（icon））

- 空一覧状態:
  - 文言例: 「まだカードがありません。新しいカードを作成して学習を始めましょう。」
  - CTAボタン: 新規作成、サンプルインポート。
- 操作フロー:
  - 検索→フィルタ→ソート→チェック→バルク操作→復習開始。UIはこの順に自然な導線を持つこと。

## データモデル（一覧で必要なフィールド）

- id: string
- title: string
- front: string（カード表面、省略可）
- tags: string[]
- collectionId: string
- nextReviewAt: ISO8601 timestamp
- lastReviewedAt: ISO8601 timestamp | null
- intervalDays: number
- easeFactor: number
- repetitionCount: number
- masteryLevel: integer (0-5)
- recentAccuracy: number (0-100)
- isArchived: boolean
- createdAt, updatedAt: ISO8601

## API（例）

### GET /api/items

クエリパラメータ例:
```text
q=<検索語>&tags[]=tag1&tags[]=tag2&collectionId=<id>&due=true&page=1&per_page=50&sort_by=nextReviewAt&sort_dir=asc
```
レスポンス例:
```json
{
	"items": [ /* Item objects */ ],
	"total": 123,
	"page": 1,
	"per_page": 50
}
```

### POST /api/items/bulk
リクエストボディ例:
```json
{
	"ids": ["id1", "id2"],
	"action": "archive",
	"add_tags": ["tagA"],
	"remove_tags": ["tagB"]
}
```

### PATCH /api/items/{id}
ボディ: 部分更新 (例: title, tags, isArchived)

### DELETE /api/items/{id}
挙動: 基本はソフトデリート。ポリシーにより完全削除を行う。

## アクセシビリティ要件
- キーボード操作: フィルタ、検索、行選択、バルク操作、復習開始が全てキーボードで可能。
- ARIA: 各行は `role="row"`、アクションは分かりやすい `aria-label` を付与。
- カラーコントラスト: WCAG AA 準拠（重要なバッジや警告は明確に識別可能）。

## ローカライズ
- 日付/時刻、相対時刻はロケール依存で表示。
- 文言管理は i18n キーで管理し、ja/en を必ず用意する。

## イベント計測（分析）
イベント例（構造はJSONで送る想定）:
```json
{
	"event": "list_view_opened",
	"payload": { "userId": "<id>", "filter": "today", "sort": "nextReviewAt" }
}
```

```json
{ "event": "item_review_started", "payload": { "itemId": "<id>", "source": "list" } }
```

```json
{ "event": "item_deleted", "payload": { "itemId": "<id>", "method": "single" } }
```

```json
{ "event": "bulk_action_performed", "payload": { "action": "archive", "count": 5 } }
```

## 受け入れ基準（Acceptance Criteria）
- AC-1: 一覧でカードが取得でき、表示列は仕様通りに表示される。
- AC-2: 検索で該当カードがヒットする。
- AC-3: フィルタ／ソート結果がAPIレスポンスと整合する。
- AC-4: バルク操作で複数カードに対して指定アクションが適用される（確認ダイアログ含む）。
- AC-5: 空一覧、ローディング、エラー状態の表示が存在し、そのUIに応じてCTAが機能する。
- AC-6: 主要なキーボードショートカット（全選択、復習開始）が動作する。
- AC-7: アクセシビリティ自動チェックで重大な違反がない（色のコントラスト、ARIA）。

## テストケース（抜粋）
- TC-1: 検索でタイトルの一部を入力し、期待するカードのみ返ること。
- TC-2: 「今日の復習」フィルタを適用し、nextReviewAt が当日以前のカードのみ表示されること。
- TC-3: 50件を超えるカードでページネーションが機能すること。
- TC-4: バルクで5件選択→アーカイブ→該当カードが isArchived=true になること。
- TC-5: 削除操作で確認モーダルが表示され、キャンセルで削除されないこと。
- TC-6: 画面リーダーで一覧行に正しいラベルが読まれること。

## 非機能的な運用上の注意
- 大量データ時のクエリ最適化（インデックス、キャッシュ）を実装すること。
- 削除はソフトデリートを基本とし、一定期間後に完全削除バッチを走らせること。
